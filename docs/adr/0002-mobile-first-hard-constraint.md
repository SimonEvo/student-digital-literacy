# 0002. Mobile-first 作为硬约束

- 状态：已接受（Accepted）
- 日期：2026-07-06
- 决策者：导师（单人）

## 背景与问题

学生的主入口是**手机扫码**（见 [0003](0003-qr-primary-access-path.md)）。多数学生年纪小、设备为手机。若按桌面页设计再缩小，窄屏下会字小、按钮挤、点击区域不足，直接劝退目标用户。

同时存在一个次要场景：导师**课上用投影 / 大屏展示**这一页。因此不能做成「仅手机」而在宽屏上崩坏。

## 决策驱动因素

- 主用户在手机上，第一行代码起就要窄屏可用。
- 目标用户年纪小 → 大字、大按钮、大点击区域。
- 次要场景：投影 / 桌面查看不能破版。

## 备选方案

1. **Mobile-first 响应式**：先为窄屏设计，向上适配宽屏（宽屏限制正文宽度即可）。
2. **桌面优先再缩小**：按桌面设计，靠媒体查询压到手机。
3. **仅手机（mobile-only）**：固定窄布局，不管宽屏。

## 决策结果

选 **方案 1：Mobile-first 响应式**（硬约束）。

具体：默认窄屏布局；大字号、大按钮、够大的点击热区；向上适配时仅限制正文最大宽度、居中，不新增桌面专属复杂布局。

### 后果

好处：

- 主用户（手机学生）体验最优。
- 投影 / 桌面查看不破版（正文限宽居中即可）。

代价 / 取舍：

- 从头就得按窄屏思维写 CSS，不能事后补 —— 这正是「硬约束」的含义。

## 方案优劣对比

### 方案 2：桌面优先再缩小

- 坏处：几乎必然产出「桌面缩小版」—— 字小、按钮挤，正是要避免的失败态。

### 方案 3：仅手机

- 坏处：导师课上投影 / 大屏展示时布局崩坏，牺牲了真实的次要场景。
