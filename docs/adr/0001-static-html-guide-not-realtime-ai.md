# 0001. 学生电子扫盲用纯静态 HTML 指南，而非学生侧实时 AI

- 状态：已接受（Accepted）
- 日期：2026-07-06
- 决策者：导师（单人）

## 背景与问题

导师反复 1:1 口头解释同一批 "book keeping" 操作类问题（表格怎么填、在线文档防丢数据、电脑基本操作、EPQ 流程）。目标：**甩掉这部分重复劳动**，课上展示一个入口，学生日后自查。

约束：

- 单人导师，无专职运维预算。
- 公司在线填表系统后端**不在导师手里**，无法改动、无法接入。
- 约 90% 的问题是**同一批重复问题**；剩余 10% 新奇 / 个性化的，导师本就得亲自处理。

问题：用什么形态承载这份指南？

## 决策驱动因素

- 成本必须与学生数 / 提问量**解耦**（单人扛得住）。
- 不得触碰公司填表系统。
- 内容常青、通用（对所有学生一样），更新频率低。
- 触达门槛要极低（学生手机扫码即用）。

## 备选方案

1. **纯静态 HTML 页 + Nginx 托管**（复用 EPQ 系统 `gantt-pro.html` 模式）。
2. **学生侧实时 AI**：学生自己提问，后端 LLM 实时作答。
3. 动态 Web 应用（带后端 / DB 的 FAQ 系统）。

## 决策结果

选 **方案 1：纯静态 HTML 页**。

AI 的角色限定为**一次性帮导师起草指南内容**（便宜、离线、网页版 AI 即可），**绝不**实时回答学生。

### 后果

好处：

- 无 per-query 成本、无 token 随量线性烧钱、无服务器扩容 → 单人可持续。
- 无需登录（不含敏感数据），无触碰公司系统。
- 部署简单，复用已验证的 `gantt-pro.html` 模式，易反悔（改一个 HTML 文件）。

代价 / 取舍：

- 无法个性化作答 → 但这 10% 本就由导师亲自处理，无净损失。
- 内容需人工维护常青 → 更新频率低，可接受（维护方式见待定项）。

## 方案优劣对比

### 方案 2：学生侧实时 AI

- 好处：能答新奇 / 个性化问题。
- 坏处：token 随学生数 / 提问量**线性烧钱**，需服务器扩容 → 单人导师成本结构上不成立。90% 重复问题用实时 AI 是浪费。

### 方案 3：动态 Web 应用

- 好处：可搜索、可后台改内容。
- 坏处：引入后端 / DB / 运维负担，对一份常青、低频更新的通用内容属过度工程。
