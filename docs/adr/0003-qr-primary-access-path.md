# 0003. QR 码为主访问路径，静态 QR + 稳定子域名

- 状态：已接受（Accepted）
- 日期：2026-07-06
- 决策者：导师（单人）

## 背景与问题

**很多学生连「把网页存书签 / 再次找到它」都做不到。** 触达方式必须极简、可反复分发，否则内容再好也到不了学生手里。

学生用手机扫码进入（决定了 mobile-first，见 [0002](0002-mobile-first-hard-constraint.md)）。QR 还刚好谐音「扫盲 / 扫码」。QR 会印在课上屏幕、发给学生的纸质 / PDF 材料上，反复出现 → 反复触达。

问题：访问路径长什么样、QR 怎么编码、是否要做可安装 App。

## 决策驱动因素

- 触达门槛要低一个量级（扫一下即进，不靠记网址 / 存书签）。
- 单人、零运维 —— 不引入需要维护的额外服务。
- QR 一旦印在纸质材料上就**无法召回**，目标 URL 必须永久稳定。
- 需要一个人也能报出口的兜底路径。

## 备选方案

分三个维度：

- **路径形态**：(a) 路径 `<域名>/help`；(b) 子域名 `help.<域名>`；(c) 独立域名。
- **QR 类型**：(a) 静态 QR（直接编码 URL）；(b) 动态 QR（经跳转 / 短链服务）。
- **加主屏**：(a) 纯网页靠浏览器原生「加到主屏」；(b) 轻量 PWA（manifest）；(c) 完整 PWA（含 Service Worker 离线）。

## 决策结果

- 路径：**子域名 `help.<域名>`**。
- QR：**静态 QR，直接编码 `https://help.<域名>/`**。
- 加主屏：**纯网页**，不写 manifest / Service Worker，靠浏览器原生「加到主屏」。
- 兜底：口头 / 文字可报出的 `help.<域名>`（学生记不住也能扫码）。

**硬承诺**：`help.<域名>` 一经印上纸质 / PDF 材料即**永久锁定，不得更改**——静态 QR 无法召回，改域名会让所有已发出的旧 QR 失效。

### 后果

好处：

- 触达门槛极低：扫码即进，无需存书签 / 记网址。
- 静态 QR 零额外基础设施、不可被第三方跳转服务卡脖子。
- 纯网页零成本，学生仍可用浏览器原生「加到主屏」得到近似 App 的入口。
- 子域名让它像独立产品，且与主域名其它服务隔离。

代价 / 取舍：

- 子域名比纯路径多一层 DNS 记录 + 证书 SAN 的一次性配置（非持续运维）。
- 静态 QR 不可改、不可统计扫码量 —— 换来零运维，符合本项目定位。
- URL 永久锁定是硬约束：域名 / 子域名策略须在首次印刷前定稿。

## 方案优劣对比

### 路径 `<域名>/help`（未选）

- 好处：零额外 DNS / 证书，最省事。
- 坏处：与主域名其它内容耦合在同一站点；独立感弱。（本可接受，但子域名的隔离与产品感更契合长期定位。）

### 动态 QR / 短链（未选）

- 好处：目标可改、可统计扫码量。
- 坏处：引入一个需维护、可能收费、可宕机的跳转服务 —— 直接违背[0001](0001-static-html-guide-not-realtime-ai.md)的零运维 / 零 per-use 成本地基。

### 轻量 / 完整 PWA（未选）

- 好处：自定义图标 / 名称；完整 PWA 还能离线。
- 坏处：对一份联网即可、低频更新的静态指南，manifest 收益有限、Service Worker 更带来缓存陷阱（学生看到旧内容）与复杂度。原生「加到主屏」已够。
